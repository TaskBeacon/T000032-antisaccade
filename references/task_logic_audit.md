# Task Logic Audit: Antisaccade Task

## 1. Paradigm Intent

- Task: `antisaccade`
- Primary construct: rule-dependent inhibitory control / response remapping
- Manipulated factors:
  - response rule (`prosaccade` vs `antisaccade`)
  - target side (`left` vs `right`)
- Dependent measures:
  - response accuracy
  - correct-trial response latency
  - timeout rate
- Key citations:
  - `W2163144416` (Geier et al., 2009, Cerebral Cortex)
  - `W2104663113` (Everling & Munoz, 2000, Journal of Neuroscience)
  - `W1875187353` (Everling et al., 1999, Journal of Neuroscience)

## 2. Block/Trial Workflow

### Block Structure

- Total blocks:
  - human: `3`
  - qa/sim: `1`
- Trials per block:
  - human: `48`
  - qa/sim: `16`
- Randomization/counterbalancing:
  - each block uses the built-in `BlockUnit.generate_conditions(...)` scheduler
  - conditions are balanced across `task.conditions = [prosaccade, antisaccade]`
  - ordering is shuffled within block
- Condition weight policy:
  - `task.condition_weights` is omitted / `null`
  - runtime therefore uses the default even weighting path
  - no task-local weight parser is needed
- Condition generation method:
  - built-in `BlockUnit.generate_conditions(...)`
  - no custom generator is required because label-level conditions are sufficient
- Runtime-generated trial values:
  - target side
  - per-trial fixation / cue / gap / ITI durations
  - correct key
  - all generated by `src/utils.py -> build_antisaccade_trial_spec(...)`
  - deterministic seed basis: block seed + trial id + condition token

### Trial State Machine

1. `fixation`
   - Onset trigger: `fixation_onset`
   - Stimuli shown: `fixation`
   - Valid keys: none
   - Timeout behavior: fixed display duration, then advance
   - Next state: `rule_cue`

2. `rule_cue`
   - Onset trigger: `rule_pro_onset` or `rule_anti_onset`
   - Stimuli shown: `rule_pro` or `rule_anti` plus `left_anchor` and `right_anchor`
   - Valid keys: none
   - Timeout behavior: fixed display duration, then advance
   - Next state: `gap`

3. `gap`
   - Onset trigger: `gap_onset`
   - Stimuli shown: `left_anchor`, `right_anchor`
   - Valid keys: none
   - Timeout behavior: fixed display duration, then advance
   - Next state: `saccade_response`

4. `saccade_response`
   - Onset trigger: `target_onset_left` or `target_onset_right`
   - Stimuli shown: anchors plus the lateral target on the sampled side
   - Valid keys: `left_key`, `right_key`
   - Timeout behavior: no response before `response_deadline` counts as timeout
   - Next state: `iti`

5. `iti`
   - Onset trigger: `iti_onset`
   - Stimuli shown: `fixation`
   - Valid keys: none
   - Timeout behavior: fixed display duration, then advance to next trial
   - Next state: next logical trial or block summary

## 3. Condition Semantics

For each condition token in `task.conditions`:

- `prosaccade`
  - Participant-facing meaning: respond toward the target side
  - Concrete stimulus realization: green rule cue (`rule_pro`) with a lateral white target on `left_target` or `right_target`
  - Outcome rules: same-side response is correct
- `antisaccade`
  - Participant-facing meaning: respond opposite the target side
  - Concrete stimulus realization: red rule cue (`rule_anti`) with a lateral white target on `left_target` or `right_target`
  - Outcome rules: opposite-side response is correct

Participant-facing text source:

- Config stimuli in `config/config.yaml`
- Summary / transition text is formatted via `stim_bank.get_and_format(...)`

Why this source is appropriate for auditability:

- visible wording stays out of code
- localization can be changed by editing YAML only
- the code only resolves condition labels into concrete stimuli and timings

Localization strategy:

- Chinese participant text is defined in config
- the task code uses stable stimulus ids and does not hardcode copy in `run_trial.py`

## 4. Response and Scoring Rules

- Response mapping:
  - `f -> left`
  - `j -> right`
- Response key source:
  - config fields `task.left_key`, `task.right_key`, `task.key_list`
- Missing-response policy:
  - no key before the deadline marks the trial as `timed_out = true`
  - timeout trials do not count as correct
- Correctness logic:
  - `prosaccade`: correct key equals the target side
  - `antisaccade`: correct key equals the opposite of the target side
- Reward/penalty updates:
  - none
- Running metrics:
  - accuracy = correct logical trials / all logical trials
  - mean RT = average RT across correct logical trials only
  - timeout count = number of logical trials without a valid response

## 5. Stimulus Layout Plan

For every screen with multiple simultaneous options/stimuli:

- Screen name: `rule_cue`
  - Stimulus IDs shown together: `rule_pro` or `rule_anti`, `left_anchor`, `right_anchor`
  - Layout anchors (`pos`): cue at `(0, 210)`, anchors at `(-340, 0)` and `(340, 0)`
  - Size/spacing (`height`, width, wrap): cue height `32`, anchor radius `34`, explicit `lineWidth: 2`
  - Readability/overlap checks: anchors remain far from the cue text and do not overlap at 1280x720
  - Rationale: clear rule cue above lateral anchors

- Screen name: `saccade_response`
  - Stimulus IDs shown together: `left_anchor`, `right_anchor`, and `left_target` or `right_target`
  - Layout anchors (`pos`): anchors at `(-340, 0)` and `(340, 0)`; target overlays the sampled side
  - Size/spacing (`height`, width, wrap): target radius `26`, anchor radius `34`
  - Readability/overlap checks: the target remains centered on the selected anchor and is visually distinct by fill color
  - Rationale: a single lateral target supports direction remapping without visual clutter

- Screen name: `instruction_text` / `block_break` / `good_bye`
  - Stimulus IDs shown together: config-defined text only
  - Layout anchors (`pos`): centered text at `(0, 90)` or config default
  - Size/spacing (`height`, width, wrap): `wrapWidth: 980`, separate text blocks for line breaks
  - Readability/overlap checks: text blocks are single-column and do not overlap because only one text stimulus is shown per screen
  - Rationale: keep non-trial messaging readable and localized in config

## 8. Inference Log

- Duration ranges (fixation, cue, gap, ITI) are implementation inferences constrained by antisaccade workflow literature rather than one fixed universal protocol.
- Human profile uses `3 x 48` trials for balanced sampling; QA/sim downsample counts for runtime practicality.
- Keyboard left/right response proxies are used in place of eye-tracker-derived saccades as an implementation inference for portability.
- Target-side and duration sampling are deterministic from `block_seed + trial_id`, which is an implementation choice for auditability and replay.
- Rule cues are implemented as colored text (`rule_pro` / `rule_anti`) instead of symbolic shapes; this preserves explicit rule communication while keeping the task portable.
